
# 開発ガイドライン

この文書には、このコードベースで作業する際に重要な情報が含まれています。これらのガイドラインに正確に従ってください。

## コア開発ルール

1. パッケージ管理
   - uv のみを使用し、pip は使用しないこと
   - インストール: `uv add package`
   - ツール実行: `uv run tool`
   - アップグレード: `uv add --dev package --upgrade-package package`
   - 禁止事項: `uv pip install`、`@latest` 構文

2. コード品質
   - すべてのコードに型ヒントを必須とする
   - 公開 API にはドキュメンテーション文字列を付与すること
   - 関数は焦点を絞り小さくすること
   - 既存のパターンに正確に従うこと
   - 行長: 最大 120 文字

3. テスト要件
   - フレームワーク: `uv run --frozen pytest`
   - 非同期テスト: asyncio ではなく anyio を使用すること
   - カバレッジ: エッジケースやエラーをテストすること
   - 新機能にはテストを追加すること
   - バグ修正には回帰テストを追加すること

- ユーザー報告に基づくバグ修正や機能追加のコミットには、次を追加してください:

  ```bash
  git commit --trailer "Reported-by:<name>"
  ```

  `<name>` にはユーザーの名前を記載します。

- Github issue に関連するコミットには、次を追加してください:

  ```bash
  git commit --trailer "Github-Issue:#<number>"
  ```

- `co-authored-by` のような表記を決して使用してはいけません。特に、コミットメッセージや PR を作成するために使用したツールに言及しないでください。

## プルリクエスト

- 何が変更されたかの詳細なメッセージを作成してください。解決しようとしている問題の高レベルな説明と、どのように解決したかに焦点を当ててください。コードの具体的な部分については、それが明確さを増す場合にのみ記述してください。

- 常にレビュアーとして `jerome3o-anthropic` と `jspahrsummers` を追加してください。

- `co-authored-by` のような表記を決して使用してはいけません。特に、コミットメッセージや PR を作成するために使用したツールに言及しないでください。

## コード整形

1. Ruff
   - フォーマット: `uv run --frozen ruff format .`
   - チェック: `uv run --frozen ruff check .`
   - 修正: `uv run --frozen ruff check . --fix`
   - 重要な問題:
     - 行長 (88 文字)
     - インポートの並び替え (I001)
     - 未使用のインポート
   - 行折り返し:
     - 文字列: 括弧を使用する
     - 関数呼び出し: 適切なインデントで複数行にする
     - インポート: 複数行に分割する

2. 型チェック
   - ツール: `uv run --frozen pyright`
   - 要件:
     - Optional に対する明示的な None チェック
     - 文字列に対する型の絞り込み
     - バージョン警告は、チェックが通れば無視してよい

3. Pre-commit
   - 設定: `.pre-commit-config.yaml`
   - 実行: git commit 時に実行
   - ツール: Prettier (YAML/JSON), Ruff (Python)
   - Ruff の更新:
     - PyPI バージョンを確認する
     - 設定の rev を更新する
     - まず設定をコミットする

## エラー解決

1. CI 失敗
   - 修正手順:
     1. フォーマット
     2. 型エラー
     3. リンティング
   - 型エラー:
     - 行の完全なコンテキストを取得する
     - Optional 型を確認する
     - 型の絞り込みを追加する
     - 関数シグネチャを検証する

2. 一般的な問題
   - 行長:
     - 文字列を括弧で分割する
     - 関数呼び出しを複数行にする
     - インポートを分割する
   - 型:
     - None チェックを追加する
     - 文字列型の絞り込みを行う
     - 既存のパターンに合わせる
   - Pytest:
     - anyio マークが見つからない場合は、pytest 実行コマンドの先頭に PYTEST_DISABLE_PLUGIN_AUTOLOAD="" を追加してみてください。例:
       `PYTEST_DISABLE_PLUGIN_AUTOLOAD="" uv run --frozen pytest`

3. ベストプラクティス
   - コミット前に git status を確認する
   - 型チェック前にフォーマッタを実行する
   - 変更は最小限にする
   - 既存のパターンに従う
   - 公開 API を文書化する
   - 十分にテストする

## 例外処理

- **例外をキャッチする際は `logger.error()` ではなく必ず `logger.exception()` を使用すること**
  - メッセージに例外オブジェクトを含めないこと: `logger.exception("Failed")` を使用し、`logger.exception(f"Failed: {e}")` のようにしないこと
- **可能な限り特定の例外をキャッチすること**:
  - ファイル操作: `except (OSError, PermissionError):`
  - JSON: `except json.JSONDecodeError:`
  - ネットワーク: `except (ConnectionError, TimeoutError):`
- **`Exception` をキャッチするのは以下の場合のみ**:
  - クラッシュさせてはいけないトップレベルのハンドラ
  - クリーンアップブロック（ログはデバッグレベルで記録する）


# 機能/非機能要件

## 機能
  
  - C言語のソースコードを読みこみ、変換して出力する。
    - **配列出力機能**
      - 配列情報を読みこみ、コメントや空白などの情報は取り除き、配列情報のみ出力する。
      - 配列名と、2次元配列以上の場合は、配列の次元情報を出力してから、配列要素を出力する。
      - 2次元配列の場合は、array[N][M] とした場合、N行M列の出力を行う。
      - 3次元配列の場合は、array[K][N][M] とした場合、K要素のN行M列出力を行う(Kは0～K-1まで)。
      - 4次元配列以上の次元の場合も、3次元配列のように、N行M列を何度も出力する。

## 非機能
-  Windowsユーザへバイナリ配布し、python環境インストールなしに使用できること。
-  コンソールツールとして使用できること。
